## A. 확정 결정 (2026-06-30)

> **파이프라인 공통 원칙 (2026-07-01)**: 각 **pipe(A6~A13)는 하나의 stage** 이며 **여러 pass 로 구성**될 수 있다. **단일책임(SRP)은 pass 수준**에서 지킨다 — pipe 가 여러 sub-task 를 묶는 것은 정상이다(예: A8 = lowering·concat/split 흡수·cost·core tag·bufferize; A9 = space alloc·dma·lifetime 각각 별 pass). 이 원칙이 **I4(concat/split 소유)·I5(A9 다중책임)** 를 함께 해소한다. 또한 **각 stage 는 그 계층에서 *생산해야 할 정보*(attr 등)가 정의되어 있고, 다음 stage 진입 전 verifier 가 "그 정보가 생산되었는가"를 검증**한다(정보 생산 완료 게이트). 최종 VLIW bitfield packing 은 A13 이지만 **그 재료 정보는 상위 stage 들이 미리 생산**하고 stage 마다 verifier 로 검증. (I8)

### A1. Front-end = onnx-mlir (확정)
- front-end 설계는 **onnx-mlir 로 확정**. 현재 방식(ONNX → onnx-mlir in-process import → ISA 근접 custom IR 로 **직접** 진입, StableHLO/IREE 미경유)을 그대로 따른다.
- ONNX 입력은 onnx-mlir 의 정적 라이브러리로 in-process import(subprocess 아님).

### A2. 입력 = int8 QDQ 양자화 모델만 (확정)
- **int8 QDQ 로 양자화된 모델만** 받는다.
- int8 QDQ 가 아니라고 판정되면 **실패 처리**한다. 단 **실패(compile fail)는 release 모드에서만** 발생한다(아래 A3 의 모드 정책을 따른다).

### A3. 두 동작 모드: release / debug (확정)
- 컴파일러는 **release 모드**와 **debug 모드** 두 가지로 동작 가능해야 한다.
- **release**: 엄격. 미지원/위반 발견 시 **compile fail 반환**.
- **debug**: 관대. 문제를 **남겨두고 error 경고를 표시하면서 컴파일은 끝까지 수행**한다.
- 이 정책은 A2(입력 판정), A6(QDQ 검증 단계)의 실패 처리에 공통 적용된다.

### A4. mono-frame / multi-frame 모드 (확정)
- **mono-frame 모드**와 **multi-frame 모드** 두 가지로 동작할 수 있어야 한다.
- **multi-frame 으로 동작하는 경우, 필요한 N/M 변수를 입력으로 받아야** 한다.
- *(정의 대기)* N/M 의 정확한 의미(프레임 수 / 슬롯 수 / 분주 비율 등)는 아직 못 박지 않음 — 구현 시 확정. 다중프레임 dyn-addr 의 residual slot-rotation(`R = frame_index mod N`) 설계는 `project_multiframe_dynaddr` 참조.

### A5. 출력 = `.ihnn` + `.json` (역할 확정, 내부 포맷 미확정)
- 출력물은 **`.ihnn` 파일과 `.json` 파일** 두 개.
- **내부 내용의 세부 포맷은 아직 미확정** — 현재 버전(`onnx_compiler_isa5.20`)의 형태를 **베이스로** 삼는다.
- **역할은 확정**:
  - **`.ihnn`** = 하드웨어에 직접 올라갈 **데이터들**(데이터 배치).
  - **`.json`** = **프로파일러와 런타임에 전달할 정보들**(예: input/output 주소·크기 등 런타임 사용정보 + 컴파일 중 생성 가능한 프로파일링 정보).

### A6. Pass pipeline 1단 = QDQ Fusion (확정, 메인 개념)
- **QDQ Fusion 을 가장 앞단의 pass pipe 로 지정**한다.
- 기본 3단계 구조(필요 시 pass 추가 가능 — 이건 메인 개념일 뿐 고정 아님):
  1. **OpInterface 주입** — 1차적으로 **하드웨어가 지원하는 op 들**에 OpInterface 를 주입한다.
  2. **변환** — 해당 OpInterface 를 가진 op 들에 대해 변환을 수행한다.
  3. **검증** — 변환 결과를 검증한다. **검증 시 지원하지 않는 op 이 있으면 compile fail** 을 반환한다(release=fail / debug=남기고 error 경고 표시하며 끝까지 컴파일 — A3).
- 구현 메커니즘은 §B1(OpInterface ExternalModel + OpInterfaceRewritePattern) 참조.

### A7. Pass pipeline 2단 = Op Fusion (확정, 메인 개념)
- QDQ Fusion 다음 pass pipe 는 **Op Fusion**.
- 목적: 우리 하드웨어 ISA 가 지원하는 op 에 맞추기 위해 op 을 **fusion** 한다.
- 경우에 따라 **분해(decompose)** 도 하지만, 우리 하드웨어가 다루는 범위는 대부분 **ONNX op 와 같은 레벨이거나 한 단계 높은 추상화**를 가지므로 보통 **raising** 을 해야 한다(예: conv-sigmoid-mul 한 번에 처리 = ONNX 보다 윗레벨 추상화 → raising 불가피). raising 은 "최대한 회피"가 원칙이나 이 지점에선 불가피함을 전제한다.

### A8. Pass pipeline 3단 = ISA Entry pipe / ISA 전용 dialect (확정, 끝까지 사용)
- Op Fusion 다음 pass pipe. **ISA 용 dialect 를 만들고, 이 dialect 를 컴파일 끝까지 사용**한다. *(이름 확정 §E: `inf_visa`/`inf_pisa`/`inf_aisa`, base `inf`.)* **(= ISA Entry pipe, 옛 이름 'ISA pipe' — A12 ISA Materialize pipe 와 entry↔materialize 짝, 2026-07-01 rename.)**
- 역할: 이전 pipe 의 **ONNX 용 op 을 하드웨어의 op 으로 맞추는** 작업.
- **dialect 내부 op = 3종**: `v-isa` / `p-isa` / `a-isa`.
  - **v-isa (virtual ISA)** — ISA 를 **먼저 받는 곳**. ISA 규칙이 엄격하지 않다. 단 그 **opcode 는 하드웨어와 통일**되어야 한다.
  - **p-isa / a-isa** — 각각 **pCore / aCore ISA 의 opcode** 들로 구성.
  - **타입 (I2 해결, 2026-07-01)**: `v-isa` 는 **tensor·memref 둘 다** 받을 수 있다(tensor→memref 경계를 **걸치는** dialect). `p-isa`/`a-isa` 는 **memref 만** 받는다(엄격). → 따라서 **`v-isa` 단계에 있는 동안 모두 memref 로 전환을 완료**해야 한다(A12 로 넘어가기 전 필수). 전환(bufferize) 지점은 아래 tensor→memref 경계 참조.
- **이번 pipe 가 하는 일 = 이전 pipe 의 op → v-isa 로 바꾸는 일련의 과정** (3 step):
  1. **Lowering** — 이전 pipe op → ISA dialect op. 이 과정에서 **concat/split(ROUTING) 은 하드웨어 구현에 맞춰 인접 op 의 `ifm/ofm_data_handling` attr 로 흡수**된다(독립 op 아님). **(I4 해결, 2026-07-01)**: 이 concat/split **흡수는 A8(ISA Entry)의 한 pass** — A6 QDQ 는 quant 경계에서 ROUTING 을 **`would_route` marker 로 표시만**(SRP: QDQ=quant/dequant 만 건드림), **실제 흡수 변환은 QDQ·Op Fusion 후 A8 visa 진입 시** 수행. (A8 = 하나의 pipe = 다중 pass, 위 공통 원칙.)
  2. **Cost model** — 입력 tensor 와 compute time 에 따른 cost model 로 **소요 시간을 attribute 로 배치**.
  3. **Core 1차 분배 (임시 tag)** — 시간을 보고 **`a-0`~`a-11` + `p-0`~`p-11` attribute(tag) 입력**(12 aCore + 12 pCore). **최대한 균등한 시간**으로 분배. **이 단계의 p/a·코어번호는 임시 tag 일 뿐 — p/a 의 엄격한 규칙은 아직 적용하지 않으며 op 은 `v-isa` 로 유지한다.** **1차 분배이며 뒤 pipe 에서 재배치**, `p-isa`/`a-isa` 로의 실제 분리(엄격 규칙 적용)는 그 뒤 pipe 에서 한다.
- **연결**:
  - step 2~3 은 §B3(CorePartition = partition-first + analytic compute+DMA cost)의 구현 지점. cost model 은 memref shape 에서 유도(full bufferize 불요), `a-*`/`p-*` 균등 분배가 "1차 분배".
  - "memref 입력 요구"는 §B2(SM/UB = memref + `#inf.*` space attr)와 정합. 단 **두 의미의 bufferization 을 구분**해야 정합: ①tensor→memref **타입 변환**(early, shape+space 확보 — 이 pipe 진입/lowering 시점, cost model·core 분배에 필요) vs ②**물리 배치**(late, SMAlloc/UBBankAlloc 의 bank/page/주소 — core 분배 *후* per-core pipe). step 2~3 은 ①만 있으면 되고 ②는 뒤 pipe.
  - p-isa/a-isa 분리는 §메모리의 "aCore 가 pCore 와 별개 ISA 가능성"과 정합.
- **상세 (2026-07-01 확정) — ISA pipe 범위·동작**:
  - **이 pipe 에서는 `v-isa` 까지만 매칭변환.** (p-isa/a-isa 로의 실제 분리는 뒤 단계 — D8.)
  - **cost model**: `v-isa` op 의 **compute 시간 + 데이터 이동** 비용을 계산. compute 시간은 **p코어·a코어 시간을 둘 다 계산**해 **더 유리한 쪽**으로 코어 종류 선택. 단 실제 p-isa/a-isa 로 옮기지 않고 **attribute 로 `p-like`/`a-like` 만 기입**(D9 tag).
  - **core scheduling**: cost 가 **균등하게 나뉘도록 12개 코어에 분배**. **지금은 대략적(rough) core scheduling 만** — 더 구체적 HW 동작이 오면 내부 scheduling 구체화. (뒤에서 materialize 되면 정확한 시간이 나오므로 그 때 상세 조정.)
  - **tensor→memref 경계 (I2 해결, 2026-07-01)**: A8 은 `inf_hnn`(tensor) → `inf_visa`(**tensor**) lowering + rough cost·코어 tag 를 **tensor shape 기준**으로 계산. 그 뒤 **`v-isa` 단계 안에서 bufferize(tensor→memref)를 완료**한다 — A9 Materialize 가 memref(+`#inf.mem`)를 요구하므로 **A8 말미 ~ A9 진입**에서 전환. 이후 A9~A11 은 `inf_visa`(**memref**), A12 가 `inf_pisa`/`inf_aisa`(memref)로 분리. `v-isa` 는 tensor·memref 둘 다 허용이라 이 전환을 한 dialect 안에서 흡수(이중정의 해소). **A8 rough cost(tensor·shape) → A10 exact cost(memref+space) 2단은 의도된 것**(rough→정밀; I1 반복 stage 가 이 2단을 수렴).

### A9. Pass pipeline 4단 = Memory Materialize pipe (memory · DMA · lifetime) — 흐름 확정 / 상세 구상 중
*(I10 네이밍, 2026-07-01: 옛 bare "Materialize" → "**Memory** Materialize" 로 prefix 부여. materialize 는 **Memory(A9) / Address(A11) / ISA(A12)** 로 부분을 나눠 진행하므로 각 prefix 로 구분한다.)*

> **주의**: 이 pipe(memory·dma·lifetime materialize, **아직 주소는 할당 안 함 = 일부(partial) materialize**)는 **HW 가 아직 개발 중**이라 정보 부족으로 세부는 확정 못 한다. **흐름만 확정**이며, 구체적 내용은 현재 구상 중이고 **바뀔 수 있음**.

- **입력 = `memref`** (A8 에서 tensor→memref 경계 넘음). **주소를 제외한 모든 데이터 할당을 여기서 명시**한다.
- **메모리 인스턴스 = 총 37개** (kind 는 sm/ub/eflash 3종): **SM 1** + **aCore UB 12** + **pCore UB 12** + **aCore eFlash 12**. (UB 는 a/p 코어별 각 12, **eFlash 는 aCore 전용** 12, SM 은 공유 1. → §B2 `#inf.mem` 을 `<kind, core-type, core-index>` 로 파라미터화 필요 — B2 보강·D11.)
- **1) 공간 할당 (space alloc)**:
  - **SM**: meta data 공간 + 계산용 공간 할당. (**eFlash 공간 할당도 이 때 같이** 수행.)
  - **UB**: **weight 용** + **input ping-pong 용** + **output ping-pong 용** 공간 할당.
  - *(코어 연산 모델)* 필요한 weight 를 UB 에 **최대한 담아두고**, 남은 공간을 분할해 input/output 을 **조금씩 가져오고**, 결과를 SM 이나 다른 코어로 나르는 방식. → 주로 **alloc/dealloc** 설정.
- **2) load/store 계산** *(아직 불명확, 빠질 수도 있음)*: 배치하더라도 **SM·UB 모두** vs **SM 만** 배치할지 개념은 **HW 정보 공개 후** 결정.
- **3) 데이터 이동 명시**: `copy`/`dma` op 사용 예상, 단 alloc 으로 할당만 할 수도. 역시 **HW 정보 후** 결정.
- **4) lifetime 기록**: 각 할당 값에 lifetime 을 **attribute 로** 기록. 데이터 **생성·소멸 사이클** 관리. **사이클 기준 = input data 가 쓰여지기 시작하는 시각**.
- **다음 단계 (A10~A13)**: A9 materialize 로 **DMA 시간까지 정확 계산 가능** → **A10 Core Equalize**(코어 배치 미세조정) → **A11 Address Materialize**(실제 주소) → **A12 ISA Materialize**(v-isa → a-isa/p-isa) → **A13 Code Generate**(.ihnn/.json).

### A10. Pass pipeline 5단 = Core Equalize pipe (확정)
- A9 Materialize 로 memory·dma·lifetime 이 배치되어 **이제 DMA 까지 포함한 시간을 정확히 계산** 가능. 이 정확한 시간으로 **연산을 각 코어에 배치하는 것을 미세조정**한다(A8 의 rough 1차 분배 → 여기서 정밀 재배치·equalize).
- 입력: memref + A8 의 `p-like`/`a-like` tag + `a-*`/`p-*` 임시 배치. cost = **정확한 compute+DMA 시간**(A9 후).
- 출력: 코어별 **균등(equalize) 배치 확정**(아직 `v-isa` 유지 — p/a 분리는 A12).
- (연동) §B3 의 "뒤 pipe 재배치" · A8/A9 의 "materialize 후 정밀조정" 이 곧 이 pipe.
- **★ I1 해결 (2026-07-01 확정) — A9↔A10 을 반복(fixed-point) stage 로**: A9 Materialize 와 A10 Core Equalize 는 **순환의존**(A10 은 A9 의 시간이 필요, A9 는 A10 의 배치에 의존)이므로 하나의 **반복 stage** 로 명시한다:
  ```
  loop:
    cost model (현재 materialization 기준)
    재배치 후보 선정
    (증분) re-materialize  +  verification (lifetime 겹침·용량·eFlash aCore 전용 등 제약)
    IF re-materialize 후 전역 cost 가 엄격히 감소 → 재배치 확정(accept)   ← 진동 차단
    ELSE 되돌림
    재배치 확정이 하나도 없으면 → 고정점 도달 → A11 로 진행
  안전망: 반복 상한 도달 시 best-so-far(유효 최저 cost) 로 종료
  ```
  - **단조 수용 규칙(명시)**: 재배치는 **re-materialize 후 전역 cost 가 실제로 더 낮아질 때만 accept**. → cost 엄격 감소 + 유한 배치수 → **종료 보장**(진동 불가; 되돌리면 cost 안 낮아지니 미수용). 도달점이 local 최적이어도 무방 — I1 은 **정합성 + 종료**만 요구(최적성 아님).
  - **비용**: full 재계산 대신 **증분 re-materialize**(옮긴 op + 영향권만).
  - **p/a flip 허용**: A10 은 코어 index 뿐 아니라 **p/a 종류도 뒤집기 가능**. **eFlash 는 SM 과 유사(공간만 분리된 개념)** 라 flip 이 진동을 키우지 않음 — verification 이 eFlash=aCore 전용 등 제약만 거른다.
  - ⇒ **A11 은 반드시 이 고정점(최종 materialization) 위에서** 주소 packing → I1(stale lifetime → 주소 aliasing → 데이터 오염) 원천 차단.

### A11. Pass pipeline 6단 = Address Materialize pipe (확정, 상세 HW 대기)
- **실제 주소를 할당**한다. §B2 `#inf.addr` attribute + SMAlloc/UBBankAlloc(C++ struct 로직 → attr 결과). 37 memory type(A9) 위에 SM page / UB bank / eFlash 주소 확정 → VLIW bitfield(`addr_ptr_weight` 18b, `addr_header` 10b) 로 이어짐.
- A9(주소 제외 materialize) + A10(코어 확정) 후라 각 데이터의 소속 메모리·코어가 concrete → 주소 배치 tractable.
- **옵션 연동(§F)**: `global_memory_alloc`(SM 단편화 최소·1프레임 전체시간 보고 배치) 이 이 단계. **`multi_frame`+`N`/`M`(I9)**: A11 이 **SM 을 코어별 구역으로 정적 분할**(구역크기 = A9/A10 의 코어별 SM footprint) + **dynamic address `base + R×stride`**(R=frame_index mod N) 부여. hazard 는 코어분할 + HW queue-complete 규칙으로 회피(FrameReplication 불요). runtime frame_index 계약 = A13 `.json`.

### A12. Pass pipeline 7단 = ISA Materialize pipe (확정)
- **`v-isa` → 실제 `a-isa`/`p-isa` dialect 로 옮긴다**(A8 에서 미룬 p/a 분리 — D8/D9 예고). A8/A10 의 `p-like`/`a-like` tag + 코어 배치를 보고 각 op 을 `inf_pisa.*`(pCore) / `inf_aisa.*`(aCore) 로 분리, **엄격한 ISA 규칙(verifier) 적용**.
- 입력: `v-isa`(공통 opcode) + 주소(A11) + 코어 tag. 출력: `inf_pisa`/`inf_aisa` op(엄격 verifier).
- **fail 계약 (I7 해결, 2026-07-01)**: p/a **legality** 를 verifier 로 검사. **a-isa fail → p-isa 로 fallback + 경고문**(pCore 가 범용 fallback). **p-isa 도 fail → 컴파일 실패(terminal)**: **debug** = 에러메시지 + 남은 부분 마저 진행 / **release** = 에러메시지 + 컴파일 중단(A3 정책과 일치). (aCore ISA 상세 규칙 = D6, HW 대기.)

### A13. Pass pipeline 8단 = Code Generate pipe (확정, 포맷 상세 미정)
- 여기까지 완성된 dialect(`inf_pisa`/`inf_aisa` + 주소·lifetime·dma attr)로 **`.ihnn` + `.json` 생산**(A5 역할). `.ihnn`=HW 데이터 배치(128B VLIW), `.json`=런타임+프로파일러 정보. 내부 포맷은 A5 대로 현 버전 베이스(D5 미정).
- **VLIW bitfield packing = 여기(A13)에서 (I8 해결, 2026-07-01)**: 128B Word0~15 비트필드 packing 은 A13 이 수행한다. **단 packing 재료 정보(opcode·act_mode·주소·scale/zp·PWL·data_handling 등)는 그 이전 상위 stage 들에서 이미 생산 완료**되어 있어야 하며, **stage 마다 "생산해야 할 정보"가 정의 + verifier 로 생산 완료 검증**(위 공통 원칙). A13 은 **이미 생산된 정보를 비트로 packing 만** 한다. (aCore ISA bitfield 상세 = D6, HW 대기.)

> **전체 파이프라인 (A6~A13, 8 pipe)**: ONNX(int8 QDQ) → onnx-mlir → **A6 QDQ Fusion** → **A7 Op Fusion** → **A8 ISA Entry**(v-isa, tensor→memref) → **A9 Memory Materialize**(memory·dma·lifetime, memref, 주소 제외) → **A10 Core Equalize**(정확 시간 재배치) → **A11 Address Materialize**(#inf.addr) → **A12 ISA Materialize**(v-isa→a-isa/p-isa) → **A13 Code Generate** → `.ihnn`+`.json`.

## B. 이 세션에서 함께 확정된 구현 결정 (A 를 뒷받침)

### B1. QDQ Fusion 구현 메커니즘 = OpInterface(ExternalModel) + OpInterfaceRewritePattern
- 타겟 op 은 **ONNX dialect op**(onnx.Conv, onnx.Sigmoid, onnx.Mul 등)이며 **우리 소유가 아니다 → onnx dialect 자체는 수정하지 않는다.**
- **trait 는 남의 op 에 못 붙인다**(컴파일타임에 op 클래스에 박히는 템플릿 인자, 런타임 부착 메커니즘 없음). → ONNX op 에는 trait 불가.
- **OpInterface 는 ExternalModel 로 op 정의 밖에서 런타임 부착 가능**. 이것이 A6-①("HW 지원 op 에 OpInterface 주입")의 실현 방법:
  - op 별 `QuantizableOpInterface::ExternalModel<Model, ONNXConvOp>` 구현(op 마다 자기 model — conv/sigmoid/mul 의 quant 의미가 달라 필연).
  - **중앙 등록 함수("list")**에서 `DialectRegistry::addExtension` 으로 `ONNXConvOp::attachInterface<ConvQuant>(ctx)` 식 일괄 부착. 새 타겟 op = 등록 리스트 한 줄 + model 1개, **pass 코드 불변**.
  - 선례: MLIR 자신이 `BufferizableOpInterface`/`TilingInterface` 등을 arith/tensor/linalg/scf upstream op 에 ExternalModel 로 부착.
- **변환·검증 적용(A6-②③)**: `OpInterfaceRewritePattern<QuantizableOpInterface>` 단일 패턴으로 **interface 가진 모든 op 에 균일 적용**(MLIR 1급 메커니즘). 매칭(DQ 벗기기·Q 확인)은 generic, op 별 차이(어떤 결과 op·scale 배치)는 model 의 `buildQuantizedForm` 에 위임.
  - interface 시그니처는 입력 구조 변이(입력 수 1 vs 2, weight/bias 유무)를 추상화해야 "동일 적용"이 깨지지 않음.
- **대안**: fusion 만 필요하면 `RewritePatternSet`(source op 종류당 패턴 1개, interface 부착 0)도 동치. 다른 pass 도 "이 op 이 quantizable 인가"를 질의해야 하면 interface 가 값을 함.
- **결정 대기**: **sigmoid 는 standalone quant op 인가, conv 의 act_mode(SiLU)로 흡수인가** — QDQ-fusion 리스트 멤버십을 가른다(현재 mock 은 SiluFusion 이 conv→sigmoid→mul → SiLU 흡수). conv-sig-mul 한 덩어리 처리(A7)와 직결.
- **arity 전략 (2026-07-01) — edge별 분류 + 제네릭 루프**: op 의 arity(1→1 sigmoid / 2→1 add·mul / 3→1 conv·fc / N→1 concat / 1→N split)를 interface 시그니처에 **박지 않는다**. 대신 **operand별 role + result별 quant 여부**를 노출하고 패턴이 그 위를 루프 → **arity = 반복 횟수로 흡수**.
  - interface 확장(기존 `getQuantizedOperandRoles` 유지 + 2개 추가): `getResultQuantized()`(result별 Q 대상 여부 — **split 1→N 커버**) + `getCategory()`(**COMPUTE**→quantized op / **ROUTING**→concat·split 은 op 아니라 `ifm/ofm_data_handling` attr 흡수). **흡수 시점(I4)**: A6 는 `would_route` marker 만, **실제 흡수 변환은 A8 ISA Entry 의 한 pass**(SRP).
  - ExternalModel 이 실제 `Operation*` 를 받아 role/quant vector 를 **런타임 길이**로 반환 → **variadic(concat N / split N) 자동 처리**. interface 시그니처엔 arity 없음.
  - **제네릭 루프**: 입력측 operand 마다 role 이 ACT/WEIGHT/BIAS 면 DQ producer require + scale/zp collect(NONQUANT skip, BIAS=NoValue 면 optional skip); 출력측 result 마다 quant 면 Q consumer require + collect. → 1→1·2→1·3→1·N→1·1→N 이 **코드 한 벌**.
  - role 이 **granularity 규정**: ACT=per-tensor, WEIGHT=per-channel(scale 배열), BIAS=int32·유도 scale(act×weight)·optional. **arity(반복)와 category(op이냐 attr이냐)는 직교**.
  - 엣지: bias 3상태(DQ/const/NoValue), mul 2nd operand const-scale(attention scale, broadcast → op-support 주의), split N출력 scale 공유, multi-use guard(result 가 Q 외 consumer 도 가지면 보류).
  - **구현**: 매칭 단계에 result-side + category 추가 + Add(2→1)/Concat(N→1)/Split(1→N) ExternalModel 부착.

### B2. SM/UB 메모리 모델링 = 하이브리드 "얇은(thin)" 형태
- SM/UB/eFlash 를 **완전 custom dialect type 으로 만들지 않는다.** `memref<T, #inf.mem<...>>` (space attr, §E)로 둔다. *(옛 flat `#inf.sm`/`#inf.ub`/`#inf.eflash` 표기는 `#inf.mem<kind, core-type, core-index, readonly, class=weight|act>` 로 통일 — line 161·D11.)*
- **space attribute 에는 "어느 메모리 + legality 를 가르는 속성(kind, readonly)"만** 올린다. region·page·bank·주소·용량 같은 **placement 는 type 에 박지 말고**, allocation pass(SMAlloc/UBBankAlloc)가 늦게 **detachable attribute 로 materialize** 한다.
- **직접 구현**: 얇은 space attr class + verifier(readonly store 금지·region∈[0,3]·page-0 회피·16B 정렬) + SMAlloc/UBBankAlloc(capacity·page/bank 배치·주소 + **weight-vs-activation 분류 분석**) + bufferize 시 space 주입(`DefaultMemorySpaceFn` 훅) + 최소 custom op(SM↔UB DMA, double-buffer swap, adaptive chunk loop).
- **upstream 공짜**: subview/view/cast 의 space 자동전파, bufferization 전체(ownership/dealloc), `memory_space_cast`, space-tagged memref 위 load/store, canon/CSE/DCE.
- **ISA 만남지점(단방향)**: space attr(kind+readonly) → alloc(region/page/bank/주소) → VLIW Word bitfield(`addr_ptr_weight` 18b, `addr_header` 10b=cluster<<8|page) 1:1. **비트필드를 high-level attr 에 싣지 말 것**(이른 객체화 = premature lowering = raising).
- **근거**: GPU(AddressSpaceAttr)·NVGPU(정수 space + memref 감싸기)·AMDGPU·IREE(placement 를 type 으로 안 만들고 affinity/bitfield attr 로) 실선례. type 에 박는 건 SPIRV 뿐이고 그건 memref 없는 emit 경계(우리론 128B-VLIW emit 경계).
- **함정**: ① bank 을 type dim 으로 만들지 말 것(alloc-time attr) ② region/page/주소를 space param 에 박지 말 것 ③ bufferization 시 모든 tensor 에 space 배정(`DefaultMemorySpaceFn` nullopt → 실패) ④ upstream transform(normalization/alloc-hoisting/layout-collapse)이 6MB hard-cap·no-spill·page-header·weight-resident 를 모르고 적용 → audit·가드 ⑤ custom load/store/alloc 만들지 말 것(HW-동사만 op).
- 상세 매핑 표(특성 15행 × 메커니즘) 및 file:line 근거는 §참조 자산의 SM/UB 워크플로 결과 파일에 있음.
- **보강 (2026-07-01, 메모리·주소 관리)**: sm/ub/eFlash 를 **별도 dialect 로 만들지 않는다**(fragmentation↑·upstream 선례 없음, GPU 도 memref+space attr). 네 메커니즘 — 어느 메모리 = **space attr**(`#inf.mem<sm\|ub\|eflash>`, dialect 아님) / 데이터 배치(tiling·padding·bank 정렬) = **memref layout**(affine/strided, load·store 가 아니라 memref TYPE 에) / **DMA = op**(memref 간 이동, layout 으로 표현·강제 불가 — layout 은 DMA 추론의 *입력*일 뿐) / 용량·뱅크·정렬 제약 = **verifier + alloc pass**.
- **주소 관리 (2026-07-01)**: **주소 전용 dialect 불필요.** ①할당 알고리즘·상태(occupancy·interval·packing solver) = **평범한 C++ struct/class**(pass 내부, IR 아님) ②배정된 주소(결과) = **typed attribute** `#inf.addr<mem, page/bank, offset>`(base `inf` dialect) → VLIW bitfield. 검증(겹침·범위·정렬) = verifier. **memref 는 논리 layout+space 만 관리, 물리주소는 관리 안 함**(런타임 descriptor 의 base ptr 은 `memref.alloc` 산물이지 정적 배치 아님). 주소를 SSA value 로 흘리지 않는 한 dialect 불요.
- **37 memory type (2026-07-01, A9 연동)**: 물리 메모리 인스턴스 = **SM 1 + aCore UB 12 + pCore UB 12 + aCore eFlash 12 = 37**. → `#inf.mem` 은 `<sm\|ub\|eflash>` 만으론 부족, **`<kind, core-type(p\|a), core-index(0..11)>`** 파라미터화 필요(eFlash 는 **aCore 전용**, SM 은 공유 1개). (D11)

### B3. CorePartition phase-ordering = partition-first + analytic cost
- **CorePartition 을 먼저** 하되, cost 를 compute-only(현 mock: MAC proxy + 28)가 아니라 **shape 에서 유도한 analytic compute+DMA** 로 올린다.
- DMA 균등화에 bufferize 불필요 — per-layer **상대** DMA cost 는 tensor shape(bytes)에서 나온다(weight 는 SM 영구상주·1회라 steady-state 제외). `cost = OVERLAP ? max(t_compute, t_dma) : (t_compute + t_dma)`.
- **bufferize 는 partition 후 per-core** 로 — 그때 각 core 의 working-set 이 concrete 해져 tractable. (core=1 bufferize 먼저 → partition 은 **비권장**: 그 DMA 가 분배 후와 불일치 + 순환의존.)
- **결정 대기**: HW 가 DMA 를 compute 와 overlap(double-buffer)하는가 → cost form(max vs sum)을 가름. micro-pipeline 6-stage 가 overlap 시사하나 확정 필요.

- **D13 (I6, HW 대기).** DMA 의 **정확한 시간** + **cost model 수식**(D3 overlap max vs sum 포함) = **HW 대기**. A9/A10 의 "정확한 시간" 표현은 HW 확정 전까진 **근사(주소-무관 성분)** 로 이해 — bank conflict·port 경합 등 주소 의존 penalty 는 A11 후에야 정확. (I1 반복 stage 구조는 유효; cost 함수만 HW 후 확정.)

## F. 옵션 체계 (2026-07-01 확정)

- **`compile-option.json`** — 대부분 옵션을 JSON 으로 관리. CLI 에는 **경로만**(`--config <file.json>`), 파서는 경로만 받고 나머지는 JSON 에서 읽음. *(사용자 제안 `llvm::opt`(=`llvm::Option`)는 clang-driver 용 table 기반이라 과함 판단 → mock 과 동일 `llvm::cl` 채택. 구조[경로만 CLI, 나머지 JSON]는 유지. 필요 시 교체 가능.)*
- **`--compile-mode` (debug | release, default debug)** — JSON 과 별개 개별 CLI 옵션. **error 대처**: `release` = error 즉시 컴파일 중단 + 에러문구. `debug` = error 방치, 에러문구만 돌려주고 끝까지 진행. (A2/A3 구체화. 입력 QDQ 판정·QDQ 검증 미지원 op 에 적용.)
- **`global_memory_alloc` (bool, default false)** — SM 메모리 단편화 최소화. **1 frame 전체 시간을 보고 가장 메모리를 아끼는 방향으로 주소 배치.**
- **`multi_frame` (bool, default false)** — 12 frame 파이프라인 동작. **프레임 간 hazard 회피 (I9 해결, 2026-07-01)**:
  - HW 규칙 — **코어의 다음 frame 은 그 코어 queue 가 완료돼야 시작** 가능. 따라서 **SM 을 코어별 구역(region)으로 정적 분할**하면 프레임 간 데이터 충돌 없음(코어별 격리 + queue-complete 직렬화).
  - **구역 크기 = 각 코어의 SM 사용량** — `core distribute scheduling(A8/A10) → mem alloc(A9) → memory scheduling(A9/A11)` 후 나오는 **코어별 footprint** 를 그대로 씀.
  - **A9/A10 의 lifetime·schedule 은 단일 프레임 그대로 유지** — hazard 는 프레임-aware lifetime 이 아니라 **SM 코어분할 + HW queue 규칙**이 담당(리뷰 I9 의 "A9 가 12-프레임 lifetime 을 만들어야" 우려를 설계로 sidestep).
  - **FrameReplication(그래프 N 복제) 불요** — multi-frame 은 **runtime frame_index**(ModelPrepare 주입) + **dynamic address `base + R×stride`**(`R = frame_index mod N`)로 처리, 컴파일 코드는 **단일-프레임**. runtime 계약은 A13 `.json`.
  - **트레이드오프 (사용자 확정)**: 정적 코어분할이 SM 최고효율은 아니나 **hazard 는 확실히 없음**. HW 안전장치가 늘면 더 효율적 방법 재고하되, 이 부분은 더 발전 없을 전망이라 **현 방식이 최선**.
  - 상세는 [[project_multiframe_dynaddr]].
- **`N`, `M`** — `multi_frame == true` 일 때만 적용(A4). *(정의 TBD = D1)*
- **원칙**: `global_memory_alloc`·`multi_frame`·`N`·`M` 은 **memory 주소 할당 단계 전용** — QDQ/Op Fusion 등 앞 pass 는 파싱·보관만, 관여 안 함.

## G. 에러 코드 규약 (2026-07-01 확정)

**동기**: 반환/에러 코드가 bare 정수(`return 1/2/3/4/5`)면 *왜* 실패했는지 알 수 없다(리뷰 **F-06** "반환코드 1 과부하"). 코드에 **의미**를 부여하고, 에러 메시지·레퍼런스 문서로 원인을 역추적 가능하게 한다.

**규약**:
1. **named `#define` 상수** — 모든 반환/에러 코드는 의미 있는 이름으로. **bare 정수 리터럴 금지**. 전용 헤더 `include/Infetron/Support/ErrorCodes.h` 한 곳에 정의(단일 출처).
2. **번호 = 도메인/단계 그룹** — 번호만 봐도 어느 단계에서 왜 났는지 안다 (exit code 는 POSIX 0–255 유지). 예시 배치(구현 시 확정):
   - `INF_OK = 0`
   - CLI/드라이버 `1x`: `INF_E_NO_INPUT`(11) · `INF_E_BAD_MODE`(12) · `INF_E_BAD_STOP_AFTER`(13)
   - config `2x`: `INF_E_CONFIG_OPEN`(21) · `INF_E_CONFIG_PARSE`(22) · `INF_E_CONFIG_UNKNOWN_KEY`(23)
   - front-end import `3x`: `INF_E_IMPORT`(31) · `INF_E_OPSET`(32)
   - Pipe1 ModelPrepare `4x`: `INF_E_NOT_QDQ`(41)
   - QDQ / OpFusion / ISA / Materialize / CodeGen 후속 단계 `5x`~`8x` (단계별 확장)
   - IO / 출력 `9x`: `INF_E_OUTPUT_OPEN`(91)
3. **에러 메시지에 코드 번호 포함** — 형식 `[INF-E41] 입력이 int8 QDQ 모델이 아님 …`. pass 의 `emitError`/`emitWarning`, 드라이버 `errs()` 모두 이 형식. 사용자가 번호로 원인 추적.
4. **에러 코드 레퍼런스 문서 (추후 필수)** — 각 코드의 **번호 · 이름(#define) · 발생 조건(why) · 대처**를 표로. 30001 문서 트리에 `errors.html`(verify 옆)로, [[skill_impl_verification_guide]] 트리에 카드 추가. **⚠️ 문서 작성 단계에서 이 항목을 반드시 떠올릴 것.**

**적용 대상·상태**: front-end(`inf-compile.cpp`·`CompileOptions`·`ModelPreparePass`·`QDQFusionPass`)
**적용 완료(2026-07-01)** — `ErrorCodes.h` 신설, 전 return/emit 을 named 코드 + `[INF-Exx]` 태그로. exit code
실측 일치(11/12/13/21/22/31/0). 리뷰 F-06 해소. **후속 단계 pass(OpFusion·ISA·Materialize·CodeGen)는
