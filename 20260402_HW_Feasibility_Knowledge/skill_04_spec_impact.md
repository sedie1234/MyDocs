# Skill: 스펙 변경 영향 분석

> **사용자**: HW 설계자, 프로젝트 매니저  
> **상황**: 하드웨어 스펙 하나를 변경했을 때 전체 시스템(컴파일러 + 런타임 + feasibility)에 미치는 영향을 한번에 파악하고 싶을 때  
> **사용법**: 변경할 스펙을 입력하면 3가지 관점에서 동시에 분석

---

## 프롬프트

```
다음 스펙 변경의 전체 영향을 분석해줘.

## 변경 사항
【예: SM을 4 MiB → 8 MiB로 확장】

## 분석 요청 (3가지 관점 모두)

### 1. Feasibility 영향
- 변경 전후 종합 판정표 비교
- FAIL → PASS 되는 항목
- 여전히 FAIL인 항목과 원인

### 2. 컴파일러 영향
- 판정 등급 (A/B/C/D)
- 수정이 필요한 코드 목록
- 선행 조건

### 3. 런타임 영향
- 적재 절차에 변경이 있는지
- 주소 관리에 영향이 있는지
- 바이너리 포맷 변경이 필요한지
```

---

## 간단 버전

```
【UB를 128KiB → 256KiB로 변경】하면:
1. feasibility가 어떻게 바뀌나?
2. 컴파일러 수정이 필요한가?
3. 런타임에 영향이 있나?
한 테이블로 요약해줘.
```

---

## 비교 분석 버전

```
다음 두 스펙을 비교해줘.

| 항목 | 스펙 A (현재) | 스펙 B (제안) |
|:---|:---|:---|
| UB | 【256 KiB】 | 【512 KiB】 |
| SM | 【4 MiB】 | 【8 MiB】 |
| 코어 | 【12】 | 【16】 |

Feasibility 결과, 컴파일러 판정, 런타임 영향을 스펙 A vs B로 대비해서 보여줘.
어떤 스펙 변경이 가장 효과적인지 순위를 매겨줘.
```
