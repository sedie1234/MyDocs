# Skill: 런타임 적재/실행 검증

> **사용자**: 런타임 개발자  
> **상황**: 런타임 구현 시 적재 순서, 주소 관리, 바이너리 파싱 방법 등을 확인하고 싶을 때  
> **사용법**: 아래 프롬프트 중 해당하는 것을 선택하여 입력

---

## 프롬프트 1: 적재 순서 확인

```
현재 설계 기준으로 런타임이 추론을 실행하기 위한 적재 순서를 알려줘.
각 단계에서:
- 무엇을 어디에 쓰는지
- 데이터 크기가 얼마인지 (【YOLOv10n】 기준)
- 실패할 수 있는 조건은 무엇인지
를 포함해서 설명해줘.
```

---

## 프롬프트 2: 바이너리 구조 확인

```
컴파일러가 생성하는 바이너리(out_bin.ihnn)의 구조를 알려줘.
런타임이 파싱할 때 다음을 분리해야 한다고 알고 있다:
1. Q records (각 pCore용)
2. Weight 데이터 (SM용)
3. aCore 프로그램
4. 메모리 배치 정보 (plan.json)

각 항목의 포맷, 크기, 식별 방법을 알려줘.
현재 구현 상태와 미구현 부분도 구분해줘.
```

---

## 프롬프트 3: 주소 관리 확인

```
SM 메모리 맵을 알려줘. 【YOLOv10n】 기준으로:
- Weight가 SM의 어디에 배치되는지 (주소 범위)
- 입력 이미지의 SM 주소
- 출력 결과의 SM 주소
- Layer 전이 시 feature map이 SM에서 어떻게 관리되는지
- 런타임이 알아야 하는 주소 정보가 plan.json에 어떤 형태로 들어있는지
```

---

## 프롬프트 4: 내 설계가 맞는지 확인

```
런타임을 다음 순서로 구현하려고 한다. 이 흐름이 맞는지 확인해줘.

1. 【내가 생각하는 Step 1】
2. 【내가 생각하는 Step 2】
3. ...

빠진 단계가 있으면 알려주고, 순서가 잘못된 부분이 있으면 수정해줘.
특히 컴파일러 출력물과의 인터페이스가 맞는지 확인해줘.
```

---

## 프롬프트 5: 자원 크기 확인

```
【YOLOv10n】을 실행할 때 런타임이 각 자원에 적재해야 하는 데이터 크기를 알려줘.

- Q: 코어당 몇 records?
- SM: weight 총 크기 + 입력 크기 + 여유?
- aCore: 프로그램 크기?

현재 HW 스펙(Q=512, UB=256KiB, SM=4MiB)에서 각각 여유가 얼마나 되는지도 알려줘.
```
